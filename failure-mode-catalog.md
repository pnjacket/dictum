# Design Decisions & Failure-Mode Catalog

Every rule in this standard exists to prevent a specific, observed failure from a real AI-driven build study or a real development-workflow gap. This catalog keeps the rationale attached to the rules, so a future editor does not delete a load-bearing rule without understanding what it held up.

## Catalog

| # | Observed failure | Root cause | Preventing mechanism (where) |
|---|------------------|------------|------------------------------|
| 1 | Frontend missing entirely; features built backend-only. | Specs layer-organized; no "feature = full slice"; the easy-to-close layer won. | Vertical-slice rule + DoD (Delivery 11.6); `SCREEN-###` binds `CAP`+`API`+`ROLE` (UX 11.9); build-ready gate = all in-scope Contract-grade (Std Part 10). |
| 2 | Frontend built; backend endpoints missing (the reverse). | No single contract both layers build against; code was the only source of truth. | Interfaces owns `API-###` contracts, work items *reference* them (11.4); owned-once-referenced + ID registry (Std Part 5). |
| 3 | Login broken though E2E "passed". | Auth was protocol-only with no journey; E2E bypassed the real login. | `E2E-STANDARD`: own code real, no bypass (Quality 11.5); substitution rule (Std Part 10a). |
| 4 | Dead nav links; orphan screens; "no story to integrate". | Navigation was prose, never a deliverable. | UX **navigation contract** `SCREEN-###/ROUTE-###` with reachability + guard (11.9); nav-wiring DoD item (Delivery 11.6). |
| 5 | Visual oddities vs the intended style guide. | The design system lived only in operator prompts. | Interim component-reuse rule + design-system `[REVISIT]` (UX 11.9). |
| 6 | Problems surfaced only at the very end (manuals). | Frontend and E2E deferred; macro-sequence hid integration gaps. | Build playbook — no deferred phases; manuals last (Delivery 11.6); build-ready gate (Std Part 10). |
| 7 | Developers blocked / unable to validate full setup (e.g. real S3). | A single production-grade DoD ignored that devs can't run real infra. | **Fidelity-staged DoD**: merge gate vs release gate (Std Part 10a); `ENV-###` fidelity map (Ops 11.10). |
| 8 | Over- or under-documenting for the product's purpose (e.g. compliance for a PoC). | No way to right-size rigor without inventing a heavy "intent" axis. | **Scope-only calibration**: concern + sub-aspect scope; Non-goals section load-bearing (Std Part 9); Baseline depth scaling (Security 11.7, Governance 11.8). |
| 9 | Contract/shape drift between concerns generally. | Each concern modeled the boundary privately. | Owned-once-referenced + stable IDs (Std Part 5); cross-concern ID registry. |
| 10 | "Tested" meaning nothing concrete. | No objective coverage definition. | Coverage map over **every in-scope contract ID** — `CAP/INV/API` + the event surface `EVT-###` + product `SUCCESS-###` — with a per-`INV` assertion (Quality 11.5; event surface generalized by #17). |
| 11 | Sensitive data tagged but never actually protected. | Classification was a fact with no enforced consequence. | facts→policy→mechanism: Domain tags → Governance `POLICY-###` → Security mechanism (11.2/11.8/11.7). |
| 12 | A contract changed; downstream docs silently kept the old shape and the build drifted. | The reference web was authoring-time only; nothing propagated a change along it. | Change events + reverse traversal + cause-attributed staleness (Std Part 10d). |
| 13 | A "Verified" doc was trusted after its upstream contract had moved underneath it. | The rung recorded a *past* verification with no signal that it had gone stale. | Staleness blocks the release gate without lowering the rung (Std Part 10d/10a). |
| 14 | A removed capability left dangling references / a reused ID pointed somewhere new. | Retirement had no precondition and IDs were reused. | Tombstones + supersession + retirement precondition; IDs never reused/renamed (Std Part 10d). |
| 15 | After build, code evolved and the docs silently lied (doc↔code drift). | Nothing watched the code side; the doc set was only ever checked at authoring time. | `drift-detector` agent (model A) emits `source: drift` change events into the lifecycle interface; adjudication routes doc-stale vs code-defect (Std Part 10d). |
| 16 | A UI control "did something weird" — Filter/Run buttons showed the user a raw framework JSON error (a 422 parse error) instead of working or failing gracefully. | Two stacked gaps surfaced building a multi-user record-management web app: (a) the real-flow E2E *reached* the screens but never **operated their controls** — least of all with the default/empty value a control submits (`entity=` for an "all" filter); (b) the error envelope wasn't **total/content-negotiated**, so the framework's own validation error bypassed it and a browser got raw JSON. | `E2E-STANDARD` **operates every screen control incl. its default state** + edge/empty inputs per `API-###` (Quality 11.5); error model is **total + content-negotiated**, with param empty-value semantics (Interfaces 11.4); error-handling pattern is total (Architecture 11.3 DD#5); **interaction/input states** + enumerated controls (UX 11.9). Sibling of #3: both are "the E2E didn't do what a real user does." |
| 17 | The **event / real-time surface** was a second-class citizen: a chat product's live delivery, presence, typing, and — critically — **subscription authorization** and **screen-reader announcement of live messages** fell through the cracks, because the downstream concerns reason in REST/click terms. | Surfaced building a real-time multi-tenant messaging SaaS. Interfaces (11.4) already defines an `event-surface` sub-type (`EVT-###`), but the *consuming* concerns enumerate only `CAP/INV/API`: Quality's coverage map omitted `EVT-###`; Security's per-interface authz was REST-only (an unchecked `EVT-C-SUBSCRIBE` would bypass tenant isolation entirely — verified: a non-member *was* able to be tested for subscribe-denial only after authz was added to the event surface); UX's state taxonomy + #16 are click/pull-centric (no connection/presence/typing/push-error states); Accessibility never required announcing live updates. | Coverage map covers **every in-scope contract incl. `EVT-###` and `SUCCESS-###`** (Quality 11.5); per-interface authorization includes the **event/subscription surface** (Security 11.7); the UX state taxonomy extends to **event-driven/ambient states** for any live UI (UX 11.9); a **live-region announcement contract** for asynchronous content updates (Accessibility 11.14). The unifying rule: where a product exposes an event surface, the Contract-grade bars that bind the request surface bind the event surface too. Sibling of #1 (a whole surface forgotten because the docs are organized around the *other* surface). |
| 18 | An invariant the docs *asserted* wasn't *enforced*, and an identifier's format chosen in one concern wasn't honored by another. | Surfaced building a real-time messaging SaaS: (a) `INV-MENTION-DELIVERY` ("exactly one notification + one email") had only a test behind it and no DB uniqueness constraint, so a concurrent re-parse could double-send — "checkable by a test" was mistaken for "enforced"; (b) Domain chose **UUID v7** message ids (for an ordering invariant) but the interface validator defaulted to **v4**, 422-ing every message-id route at runtime — a layer-drift the ID web didn't catch because an id's *version/format* isn't part of the `ENTITY-###` id contract. | A uniqueness/"exactly-once" `INV-###` must **name its enforcement** — a DB constraint / transaction, or a **structural / by-construction** mechanism where the store has no declarative constraints (a fixed primary key, a single-record `put()`, a transaction scope: the store *cannot represent* the violation) — not merely a test; an identifier's **format (incl. UUID version) is part of the id contract** Domain owns and Interfaces' validators must honor (Domain 11.2; Interfaces 11.4). A derived key an invariant depends on (e.g. a canonical DM key) is itself a contract to specify. |
| 19 | An endpoint specified as "→ `ENTITY-X`" forced every layer (service, client, test) to **invent the wire shape** — the same drift Dictum prevents at the ID level, re-appearing one level down at the field level. | Surfaced building a real-time messaging SaaS, independently by three implementers: "returns `ENTITY-WORKSPACE`" / "`ENTITY-MEMBERSHIP[]`+user" / "messages + channels" gave no field-level projection (which fields, casing, whether `email` is exposed). The message/conversation surfaces that *did* carry an explicit DTO built cleanly; the rest diverged. | Interfaces' Contract-grade bar requires a **concrete response projection per element** — the fields exposed and their names, not just the owning `ENTITY-###` (Interfaces 11.4). The entity is the *source*; the wire projection is the *contract a client builds against*. |
| 20 | A modal's scrim blocked pointer interaction with the background, but **Tab walked straight past it** — keyboard users could operate the supposedly-inert background. Surfaced building a client-only real-time browser game, and *only because Accessibility happened to be in scope*; with a11y scoped out it ships as a plain correctness bug. | Modal integrity was split by **input channel**: UX contracted the pointer scrim while keyboard inertness was implicitly parked in Accessibility (11.14) — an *optional module* — so half the behavior lived in a concern that may legitimately be absent. | The UX modal bar requires **input-channel-complete inertness** — pointer AND keyboard — as part of Contract-grade UX itself; 11.14 owns only focus-order refinement (UX 11.9). Sibling of #16/#17: a bar carved along the wrong axis lets half the behavior escape. |
| 21 | The release gate's binding environment (`ENV-###`: display, hardware GPU, branded browsers) was **structurally unavailable on the build host** (headless VM, virtual VGA); the gate items had nowhere to go but a fake pass or silent disappearance. Surfaced building a client-only real-time browser game. | Part 10a said "deferred and tracked, never silently skipped" — but deferral implies a scheduled future run; "the binding environment cannot exist here, and no run is scheduled" had no terminal state. | **Blocked-by-environment** is a first-class gate-item state = open deferral **+** recorded accountable waiver (cause / scope / re-run condition), optionally accompanied by **off-gate evidence runs** at the highest available fidelity, artifact-tagged non-binding (Std Part 10a). In the build that surfaced this, the waiver was recorded (WVR-001), software-GL evidence archived, and the capability honestly left **not** Verified. |
| 22 | The v0.11 **ambient-states bar read as inapplicable**: it triggered on server-pushed `EVT-###`, while a client-only real-time browser game had exactly the failure-mode-#17 states — an autosave-timer toast, a lock-loss pause, worker-result banners — with no server in sight. The bar nearly excluded the very states it was written to catch. | A rule was written against the **stack** that surfaced it (server push) instead of the **behavior** it binds (any state change not produced by a user action); every differently-shaped product must then argue itself back into the bar's letter. | **Generalize by behavior, not by stack:** a bar triggers on the behavior, with the original transport as one *instance* — the ambient bar on any non-user-action state change (UX 11.9; announcement contract 11.14). The same umbrella covers the client-only adaptation notes in Architecture 11.3, Interfaces 11.4, Quality 11.5, and Performance 11.13. |
| 23 | An error row's gate tier demanded an **E2E proof no documented forcing could produce**: the *Forced-by* cell named only unit-level forcings (stub worker, fake timers) while the row was owed an end-to-end "forced condition → user-visible surface" check — discovered at build time, forcing an unplanned doc edit mid-slice. | The forcing column and the proving tier were authored at different times against different mental models; nothing required them to be jointly satisfiable. | An error-catalog row's **Forced-by cell and its proving tier are authored together** — the named forcing must be usable at the tier that proves the row (Interfaces 11.4). |
| 24 | Frame-time percentile targets were **untestable as written**: no sample window, no disposition for the trailing partial bucket, no statement of where "median of 3 runs" gets its 3 samples, no re-run rule — the difference between a testable and a decorative `PERF-###` row. | A percentile is a statistic over a **population the row never defined**; each unstated method detail is a place the doc and the evidence can disagree. | A percentile target states its **sample population, window, re-run rule, and method details** (partial-bucket disposition, sample source), co-owned with Quality's flake/re-run policy (Performance 11.13; Quality 11.5). |
| 25 | With a **human operator present**, intake still ran in its unattended-inference posture — took a thin brief, **inferred the rest, and proceeded**: a job-scheduling doc set settled a job lifecycle with no cancel/abort state and deferred retry-and-undo of already-dispatched side effects (`CAP-UNDO`), **none of it ever asked**. The operator found out only by auditing the generated docs; whole features were one assumption away from silently never existing. | The autonomous-mode concession (infer-and-mark, the price of running the validation builds unattended) **leaked into interactive use** as the default. "Adaptive interview, not a fixed questionnaire" + "only ask what you can't infer" were read as *minimal* elicitation, conflating adaptiveness (which questions are relevant) with thinness (how few to ask) — so the model stopped at "inferred enough" instead of at requirement saturation. Dictum is interview-first; that posture was simply never engaged. | **Two elicitation modes, distinction load-bearing** (Std Part 0.6, Part 10): **interactive** (operator present — the default) drives to **saturation** in coverage-map-driven rounds (concern sub-aspects × each capability's lifecycle/edge/failure/NFR), never infer-and-proceed; **autonomous** (unattended) is the only mode that substitutes inference, and then every inferred answer is an `[ASSUMPTION]` surfaced for audit. The interviewing skills default to interactive and switch to inference only when explicitly unattended (`doc-scaffold`/`doc-levelup`/`doc-feature`). Sibling of #8: #8 right-sizes *breadth of scope*; #25 ensures that *within* scope, an operator-present run asks rather than assumes. |
| 26 | Applying Dictum to an **existing, already-built repo** stalled: the whole codebase read as one undifferentiated mass of **code-ahead drift**, and the greenfield tooling couldn't orient — `doc-scaffold` is deliberately code-blind (never assume the product from the repo), while `drift-detector` (10d) / `implementation-planner` (10e) are both binding- and rung-gated over a doc set that didn't exist yet, so neither could **bootstrap** docs from bare code. The operator either hand-wrote everything or let the model free-associate a doc set that silently invented intent the code never proved. | A real development-workflow gap: the method had a **greenfield entry only**. Two code→doc directions existed (drift maintenance, forward implementation) but not the **code→doc bootstrap** they both presuppose; and the recoverability distinction — code proves *structure*, never *intent* (why, non-goals, threat model, SLAs, business) — was nowhere named, so a naive reverse pass would either omit the structure or fabricate the intent. | **Reverse-authoring flow** (Std **Part 10f**): the `code-cartographer` agent reverse-maps the repo (traits + candidate contracts + draft binding map + a **recoverability report**) and the `doc-excavate` skill authors an as-built baseline via a **confirm-and-fill interview** — code sources structure, the interview sources intent (non-goals/scope-out kinds categorically un-extractable, Part 9); every inference is an `[ASSUMPTION]` until confirmed, smells become `[REVISIT]` (never enshrined as contracts), the binding map is **minted at authoring time** (resolving the code-ahead baseline into the fixpoint the drift tools need), and Verified stays Delivery's. Produces a Sketch/Specified baseline, not a build-ready set. Sibling of #25: #25 keeps an operator-present *interview* from degrading to inference; #26 gives *reverse-authoring* the same discipline — extract what code proves, interview the rest, never guess intent. |
| 27 | Reverse-authoring a real brownfield repo, the biggest hazard wasn't missing structure — it was **structure without behavior** silently promoted to a delivered contract: an orphan schema table nothing reads or writes, a README-claimed feature (an analytics rollup, an audit trail) with **no implementing code**, a UI call to an endpoint the backend never defines. A naive reverse pass mints these as clean contracts (or worse, as `Verified` because a *type* or *table* exists) — documenting **vaporware as built**. Surfaced across the Part 10f validation repos, where cartographer runs independently reported this as the single dominant signal on messy codebases. | Extraction confidence was pinned to *structural* evidence (a schema table, a README line, a typed call site) with **no reachability check** back to behavior; and the method had **no bucket** for "structure present, behavior absent" — so it fell between "delivered contract," "smell," and "confident capability" and got smuggled into whichever was nearest. It is in fact the concrete, contract-level instance of #8/Part-9's *deliberately-excluded-vs-not-built-yet* ambiguity, which **code cannot resolve**. | The `code-cartographer` runs a **reachability check** (an entity/route with no reader/writer own-code locator is `partial`, not `high`) and emits **scaffolded-but-unbuilt ("aspirational") contracts** as a first-class bucket, never in the delivered inventory; `doc-excavate` interviews each to **`deferred`** (`[FUTURE-SCOPE]`, re-entry owed) / **`absent`** (dropped, dead structure a `[REVISIT]`) / **defect** (behavior owed, a `[GAP]`), mints **no behavioral binding** for it (a coverage gap by construction — the forward tools' "build this" signal), and **never** claims Verified from structure alone (Std **Part 10f**). Sibling of #26: #26 gives reverse-authoring an *entry*; #27 keeps that entry from documenting intent as reality. |
| 28 | A reverse-authored doc set adopted **onto an existing doc web** silently **double-owned** contracts: the excavation coined *fresh* IDs from code names (a prior `CAP-001` re-minted as `CAP-QUERY`) and **over-fragmented** (one authored contract surfacing as several finer code-level ones), so the same real contract now existed under two IDs at two granularities — quietly breaking owned-once and inflating the traceability web. | Reverse-authoring recovers *structure* at high fidelity but **cannot recover a contract's stable *identity*** unless it was written into the code: an un-annotated pass mints a **new ID space** with no shared key to the incumbent web, and reads code at a finer grain than a human groups it. Observed in a regeneration-fidelity control (write-up in the research companion): structure recovers comparably without annotations, but exact-ID recovery collapses and auto-alignment to the existing web drops from mechanizable to manual. | Two moves (Std **Part 10f**): **(1)** a recommended **build-time in-code ID annotation** convention (the `DICT: <ID>` marker token in a host-language comment at the code site — only the token grammar is fixed, never the comment leader, since the standard does not control language choice) that turns a later code→doc pass from a fresh *mint* into a *recovery* of the author's own IDs — cheap while building, and it also sharpens the drift-detector; **(2)** a mandatory **ID-reconciliation step** before adopting any regenerated set onto an existing web — join to incumbent IDs, emit a **tombstone-alias table** (never reuse/drop an ID, Part 10d), **roll up** over-fragmented contracts to the incumbent granularity, and flag unmatched for adjudication. First-bootstrap onto an empty web owes none of this. Structure is annotation-independent; **identity is not** — annotations buy naming, not knowledge. |
| 29 | A reverse-authored doc set recovered the *nodes* (entities, endpoints, invariants) but emitted them **flat and unlinked** at raw code granularity — dozens of separate endpoint contracts with no capability grouping and almost none of the `CAP→SCREEN→API→ENTITY` reference web. It "found everything" yet was un-navigable for a human and un-traceable for the method: the traceability web the whole standard exists to provide (owned-once, referenced-everywhere) was absent, and the map's real value — the *edges* — was missing. The tempting fix (roll contracts up to match the hand-authored count) is wrong: the author's coarser grain encodes **intent** (which consumer, which grouping) that code can't recover, so collapsing is lossy *and* a guess — and for an AI implementer the finer grain is often better. | Extraction was **node-first**: it enumerated contracts at the grain the code presents (one per symbol/route) and treated "did we find each thing" (recall) as the whole job, with no pass to reconstruct the **edges** (which the call graph / imports / data flow largely *do* carry) or to synthesize a coarse index. "Granularity" was being treated as a single lossy choice rather than a controlled view over a linked hierarchy. | **Granularity is a controlled hierarchy, not a roll-up** (Std **Part 10f**): keep contracts at the **fine** code grain by default (explicit context; independently drift-checkable) **and synthesize a coarse `CAP`/resource index over them**, wired by a **reference web reconstructed from code** (call graph / imports / data flow) — owned-once at each level, cross-linked, so a manifest `authoring_granularity` selects the primary view and drift/reconciliation key on either grain. The `code-cartographer` extracts **edges as well as nodes** and proposes groupings; `doc-excavate` authors the linked hierarchy (extract → synthesize), never a flat dump. The defect to avoid is *unlinked*, not *many*. |
| 30 | Independent validation-build doc sets reached build-ready — one of them fully **built**, through a series of enhancement deltas — with every doc still `status: draft` and build markers unstripped, while the derived README claimed "build-ready." The draft/published signal had become noise: a consumer could not tell a handoff-grade set from an abandoned work-in-progress, and the set's own lifecycle metadata was lying in exactly the way the standard forbids product docs to lie. | The publish step existed only as prose (Part 6); **no gate consumed `status:` and no tool checked it** — an assertion with no observable check, exempted from the standard's own first principle. Publish also had **no owner** (every other step in the flow has a named skill/agent/gate) and **no recurrence rule** under the enhancement lifecycle, so even a once-published set would decay at its first delta. | Publish is **gate-bound on authoring events only** (Std Part 6): the build-ready gate hands off a *published* set (Part 10); a material edit **re-drafts** the touched doc and the delta's **hand-off re-publishes** it (Part 10e) — implementation events touch build-status/staleness, never `status:`, preserving doc maturity ≠ implementation status. The `doc-maturity-auditor` detects **status↔claim inconsistency** from implementation *evidence* (a named check — proof a gate was skipped, not a status coupling); `implementation-planner` warns on draft input; `doc-feature` re-drafts on edit and re-publishes at hand-off. Sibling of #13: both keep a stateful lifecycle signal trustworthy by attaching it to a gate instead of to authorial discipline. |
| 31 | One logical DTO realized **twice** — a typed backend class and a hand-written typed SPA interface for the same wire object — **drifted in shape and name**: a field renamed on one side, an enum represented as ordinals where the other side parsed names, a date emitted in a format the consumer never expected. Each side type-checked green in its own language; the break surfaced only at runtime, at the JSON boundary. Surfaced building a small full-stack web app with a typed backend and a typed SPA. | The binding map indexed the contract's realizations without distinguishing the **producer** end from the **consumer** end, and the **serialization convention the two sides meet on** — casing, enum representation, date format — was contracted nowhere: each language's realization was checkable only against itself, never against the shared wire form, so two green type-checkers happily disagreed about one contract. | **Dual-realization binding** — an optional binding-map form (`templates/binding-map.template.md`; Std Part 10d): per contract, **producer + consumer locators** plus an explicit **wire-serialization sub-contract** (casing / enum representation / date format); drift is then checkable at *both* ends against the declared wire form. Sibling of #19: #19 pins the wire shape in the *doc*; #31 keeps two *code* realizations of that shape independently checkable against it. |
| 32 | After re-vendoring a **newer standard version**, every older doc set audited had **silent partition holes**: a sub-aspect key the newer version had published appeared in **neither** the manifest's `out_of_scope_subaspects` **nor** the doc's in-scope readout — the in/out scope decision that key demands was never made — and in each set the new key's activating trigger was in fact firing (the subject existed in the product). | The sub-aspect vocabulary is published by the standard (Part 9), but the *partition* over it is recorded per product — and a vocabulary addition ships with **no event on the product side**: nothing forces the new key through a scope decision, and partition checking read "not listed out" as "in scope", silently. A standard upgrade thereby strands previously conforming sets without any of them changing a line. | Three mechanisms, one per surface: the **versioning policy** (RELEASES.md) requires a MINOR release's notes to **name each addition's re-conformance step**; the `doc-maturity-auditor` runs a named **vocabulary re-partition check** (every published key of every in-scope concern must be accounted for in the set's recorded partition — a hole is an error, attributed to the version delta where the authored-against version is recorded); and the **upgrade walk** (`install-dictum` re-vendor path / `doc-levelup`) records an explicit in/out decision per new key at upgrade time — an operator interview, never a silent default. Sibling of #30: a lifecycle claim ("the partition is total") stays trustworthy only while a named check consumes it. |
| 33 | Rebuilding a documented product from its doc set alone, across a MINOR standard upgrade, the named re-partition check (#32) **ran clean and passed a defective set**. The upgrade had added no sub-aspect *keys* — its additions were a new owned contract *kind* (under an existing key) and a *broadened concern trigger* — so the key-diff had nothing to catch, while the set left the new kind unminted (stated as prose) and still scoped a now-activating concern out on its pre-upgrade reasoning. The headline upgrade-safety check was a no-op on precisely the release that introduced it. | The re-partition check was defined **over sub-aspect keys only**, but a standard's vocabulary changes on three surfaces: keys, **owned contract kinds**, and **concern triggers**. A kind added under an existing key, or a broadened trigger, changes what a conforming set must reconcile **without changing the key set** — invisible to a key-diff. | The `doc-maturity-auditor`'s re-partition check reconciles **all three** per in-scope concern: keys (as before), **owned contract *kinds*** (a kind the vendored spec now owns but the set states only as prose → `unminted-kind` error) and **concern *triggers*** (a broadened trigger that now fires while the manifest still declines the concern → `stale-scope-decision` error). Kinds/triggers are prose in the specs, so this is a **model-A judgment** step; a deterministic backstop waits on the specs exposing kinds/triggers as machine-readable fields (research companion). Direct extension of #32: #32 catches the key hole; #33 closes the kind/trigger holes a key-diff is blind to. |
| 34 | Across a set, roughly a set's worth of owned IDs appeared on **more than one** register-*shaped* line — a `Contracts` recap of `Requirements`, or one concern's accountability table carrying an ID another concern owned — so a conforming extractor saw several candidate "definitions" for one ID and could not decide which line defines it. | Part 5 fixed the register *form* and declared every non-mint occurrence "non-minting", but a recap/reference table is, by shape, a register line — and the bar supplied **no rule keeping "exactly one register line per ID" true** when Part 4 itself blesses recap tables. "Non-minting" was unambiguous in principle and undecidable in practice. | Part 5 keeps one register line per ID **by construction, order-independently** (an order-dependent "first line wins" was rejected as fragile): the mint is in the **ID's owning-concern doc** (a register-shaped line elsewhere is a reference by definition), and a **recap/reference table must not place an owned ID in register-form position** — a second register-shaped line for one ID is an **authoring error the auditor flags**, not an ambiguity to resolve. Sibling of the register-form bar: the bar makes a definition line locatable; this keeps it *unique*. |
| 35 | An unattended, doc-led build ran many slices to green (bindings resolving, coverage passing) while the Delivery-owned **build-status record was never updated** — it still declared every slice unbuilt after the whole product was built. The code was fine; the *record of what was built* was stale, and only a late release-readiness pass noticed. | Part 10e stated that loop-closure "records the slice Verified" as a **consequence**, not an **enforced exit criterion of the slice**. A human updates the record by habit; an autonomous builder optimizing for "tests green" has no habit and no gate, so a bookkeeping step with no test that fails when skipped silently didn't happen. | Two-part fix that preserves doc maturity ≠ implementation status: Part 10e **names recording build-status an explicit slice exit criterion** (the normative half), and the **detection** — built-but-record-unbuilt, an evidence-based decidable check — lives with the **deterministic (model-B) checkers in the research companion**, deliberately **not** in the `doc-maturity-auditor` (whose remit is doc maturity, not build evidence). Sibling of #30: #30 catches a *doc-status* claim outrunning evidence; #35 catches a *build-status* record lagging behind it — each without coupling the two. |

## Key design decisions (cross-cutting)

| Decision | Rationale |
|---|---|
| Two axes only (breadth × depth); scope is the sole calibration lever. | Avoids a third "intent/rigor" axis; one mechanism right-sizes both CLI↔platform and PoC↔production. |
| Contract-grade is a fixed, absolute bar; PoC-ness comes from a smaller scope. | Keeps "build-ready" meaningful everywhere; you never ship a weaker bar, only a narrower surface. |
| Doc maturity ≠ implementation status; Verified is owned by Delivery. | Lets documentation lead implementation without conflating "specified" with "built". |
| Enforcement is advisory (docs + tooling that warns, never blocks). | Chosen posture; keeps the method portable and low-friction. The merge/release gates and scope warnings inform, the human decides. |
| Self-exemplifying standard + extractable schema. | The standard is its own worked example, and the machine-critical parts (catalog, rubric, manifest) are tool-readable. |
| Enhancement lifecycle reuses the ID web as a change-impact graph (derived reverse traversal), not a separate change-tracking system. | The same owned-once web that prevents *authoring* drift prevents *enhancement* drift; a parallel copy would itself drift (Part 0.5, Part 10d). |
| Staleness is orthogonal to the rung — it keeps the rung but blocks the release gate. | Preserves the record that a doc was once verified while still preventing a ship against a contract that moved underneath it; staleness is a separate, clearable signal. |
| A stored binding map is allowed for drift detection, though stored edge lists are otherwise forbidden (Part 0.5). | It records *underivable* doc↔code correspondence (not a copy of derivable data) and is self-validating each run, so it can't silently rot — a dangling binding is itself a finding. |
| Sub-aspect keys are an enumerated vocabulary (kebab-case from each concern's *Sub-aspects* table), not free-form manifest strings. | Surfaced by a CLI validation run: ad-hoc `out_of_scope_subaspects` strings are uncheckable and hide typos/uncatalogued exclusions. An enumerated vocabulary makes every scope decision validatable. |
| A fully-scoped-out concern is Contract-grade on its asserted residual, with no contracts/acceptance section. | A concern whose every testable sub-aspect is out of scope has nothing testable left; demanding a contracts section would be busywork and would tempt a weaker bar. Scope stays the only lever (Part 9). |
| An external tracker is a downstream mirror of what Dictum owns, keyed on the ID web (contracts + `Verified`/build-status repo-owned; demand and triage may live tracker-side but are never built / written to the repo without a Contract-grade ID / a reproduction). | Extends owned-once and no-stored-edges across the repo boundary; keeps a ticket from becoming a shadow contract (#2) and a suspected bug from polluting the oracle before it's real. A bug moves build-status, not the contract — a falsification hypothesis, not a rival truth (Std Part 10c, 11.6 DD#11). |
| `published` is a lifecycle state bound to gates — not a sixth rung, and not a new tool. | Publishing adds no knowledge, so it cannot be a rung (#30); it is ~three mechanical edits, so a dedicated skill would be ceremony to fix ceremony. Binding it to the authoring gates the flow already passes (build-ready, delta hand-off) and to the checks the existing tools already run is the same enforcement pattern the standard applies to every other claim — while keeping publish an authoring event, never an implementation one (Std Part 6, 10, 10e). |
| A dangling binding is a `binding-stale` finding, separate from code↔doc drift directions. | The binding map is a third artifact; conflating its rot with a code-vs-doc divergence mis-routes the fix — you repair the index, not the doc or the code. |
| Forward references use a "described-here, minted-by-X" placeholder, tightened when the producer reaches Contract-grade. | Surfaced twice in one validation build: dependency-ordered authoring still leaves seams where a consumer needs an ID its owner hasn't minted; minting from the consumer would break owned-once, and tightening the by-description binding later is a sanctioned seam, not drift (Part 5). |
| A contract may bound a delegated value without fixing it (contracts-as-bounds). | Recurs wherever a11y/perf constrain design-delegated choices (a min contrast ratio over delegated color tokens; a frame budget over implementation-chosen algorithms); naming the pattern keeps the bound testable without mistaking it for over-specification (Part 5). |
| Scope-out has two kinds — `absent` (subject doesn't exist; justification = the trait fact) vs `deferred` (calibration; re-entry note owed) — both recorded under the published key. | Whether a `[FUTURE-SCOPE]` re-entry note is owed depends on the kind; leaving the distinction unnamed either litters absent subjects with re-entry notes or silently drops deferred ones (Part 9, manifest Part 7). |
| Dictum is interview-first; inference is an **autonomous-mode** concession, not the doctrine. Interactive runs elicit to saturation; only unattended runs infer, and then mark every inference `[ASSUMPTION]`. | The standard's validation builds were infer-heavy because they ran *unattended* — a property of those drives, not of the method. Naming the two modes stops the autonomy concession from leaking into operator-present use as "ask as little as possible" (Part 0.6; failure #25). |
| An owned ID is minted on a **register line** of fixed form; every other occurrence is a non-minting reference (Part 5). | Every conforming set already minted IDs on one recognizable definition-line idiom, but the standard never stated the form — so deterministic tooling had to *guess* where an ID is owned, and a definition-like mention could masquerade as a second mint. Fixing only the form (never the content, the shape choice, or any tool) turns the existing idiom into the method's second machine-extraction extension point, alongside the in-code `DICT: <ID>` token (Part 10f). |
| The **ID token grammar is stated**, not just claimed: `[A-Z][A-Z0-9]+(-[A-Z0-9]+)+` (Part 5). | The standard declared "only the token grammar is normative" for the `DICT: <ID>` marker — and then never wrote the grammar down. The first deterministic tool had to invent its own pattern, and a second independently built tool could disagree at the margins (case, underscores, wildcard tokens), silently fragmenting the very extension points that exist so tools *don't* guess. Both extension points presuppose one token shape; the standard now states it once, and detection tooling may scan looser only to flag near-misses, never to mint. |

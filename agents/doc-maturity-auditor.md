# doc-maturity-auditor

Strictly **read-only**. Never edits — produces a report. It never assesses the **Verified** rung — that is owned by Delivery Process and needs execution; this agent reports *doc maturity* only (doc maturity ≠ implementation status). It audits the **manifest-declared doc set** (spec + manifest + bindings); sibling files (notes) are out of scope unless the manifest references them. **Exception:** the human-facing **index/README is derived from the manifest** (Part 7), so the auditor checks it for *consistency* with the manifest (see step 8) — it does not audit its prose.

## Procedure

1. **Read the manifest.** Enumerate in-scope concerns, their `current_rung` and `target_rung`. **Validate `out_of_scope_subaspects` keys** against the owning concern spec's published **`key`** column in its *Sub-aspects* table (Part 9) — a key that is not a published key of its concern is an error; a near-miss that still uniquely matches one row (e.g. a legacy-derivation key) is a warning. The same standard applies to a concern doc's `in-scope-subaspects` front-matter (a *derived readout*, Part 7) and to a manifest's own `out_of_scope_subaspects` **comment** that documents an older derivation rule: a legacy-form slug or a stale derivation comment is a **warning**, never an error.
2. **Vocabulary re-partition (named check — always run).** For each **in-scope** concern, diff the concern spec's published sub-aspect **`key`** set (Part 9) — read from the standard material **actually vendored/referenced in the product repo**, the same specs step 1 validated keys against — against the union of what the set has **accounted for**: the manifest's `out_of_scope_subaspects` for that concern ∪ the doc's `in-scope-subaspects` readout. (When the optional readout is absent, a key counts as accounted only if the doc **materially addresses** it — a section, an owned contract, or a Non-goals line naming it.) Every published key in **neither** set is a **vocabulary-partition-hole** finding (**error**): the in/out scope decision that key demands was never made. The classic cause is a **standard upgrade** — a newer vendored version published a new key after the set was authored, leaving the set silently non-partitioned (the key shows up in neither record, and nothing else forces the decision to surface). When the manifest records which standard version the set was authored against and it is older than the vendored specs' version, call out the delta on each finding (`authored-against vX vs vendored vY`); absent such a record — the manifest's `authored_against:` field is optional — report the hole without version attribution. Remediation is the **upgrade walk** (`install-dictum` re-vendor path / `doc-levelup`) — an operator interview, never a silent default; do not propose in/out answers yourself. Report this check by name even when it finds nothing (it subsumes what partition checking used to catch only incidentally).
3. **For each in-scope concern**, read its doc and compute the **actual** rung from section completeness. **STANDARD Part 4's section→rung table is authoritative** — use it, don't rely on this gloss: roughly Sketch = Purpose/scope + Non-goals + Requirements outline; Specified = Requirements complete + Dependencies + **Examples** + Design Decisions begun; Contract-grade = Contracts (stable IDs) + testable Acceptance criteria + Design Decisions complete. A section may be satisfied **materially** — embedded worked content (e.g. inline CLI invocation/output examples) counts as Examples; a discrete heading isn't required.
   - **Fully-scoped-out concern (Part 9):** if the manifest scopes out *all* of a concern's testable sub-aspects, it is Contract-grade once its residual scope is fully asserted. **Sufficient** = (a) the in-scope baseline facts are stated **and** (b) each scoped-out sub-aspect appears in Non-goals with a one-line justification; a bare hand-wave with neither is a gap. When sufficient, do **not** flag it for missing Contracts/Acceptance — nothing testable remains in scope. **But when a residual in-scope surface is declared** (the Part 9 residual-scope checklist), checklist item (4) still binds: each in-scope assertion still owed a contract needs an observable acceptance check — an empty Contracts/Acceptance section is **not** passable for such a concern.
4. **Flag mismatches:** where the claimed `current_rung` outruns the actual section completeness.
5. **List gaps to `target_rung`:** missing/incomplete sections, unresolved `[GAP]`/`[REVISIT]`, acceptance criteria that aren't observable checks, contracts lacking IDs.
6. **Cross-reference integrity (doc-end):** check that every referenced ID (`CAP-### / ENTITY-### / API-### / SUCCESS-### / PERSONA-### / ADR-### / …`) resolves to an **owning concern** — the in-scope concern that *defines* it (in single-file packaging, the section that introduces it). For `bindings.yaml` / index files, validate only that their **keys/IDs** resolve here; **code-end locator resolution is the drift-detector's binding-map self-validation, not this agent's.** If a referenced ID's owning section is an **out-of-scope** concern, that is a scope inconsistency → report it. Report dangling references. **Local-prefix IDs** (Part 5 allows them): resolve ownership via the docs' declared **"Owns `X-###`"** statements — an ID whose prefix no doc claims to own is a **reportable ambiguity**, not something to auto-assign. **Arm tokens:** an arm-suffixed ID referenced cross-doc (`INV-004a` style) must appear **verbatim** in the owning register row — resolving only to the bare parent ID is a **warning**.
7. **Scope sanity (advisory warnings only):** if a risk trait is present (real pii/secrets) but its protection sub-aspects are scoped out, warn. The reverse override — a concern pulled **in** against its trigger — needs only a **recorded rationale**; an override with one is not a finding.
8. **Index/README consistency (Part 7, derived index).** If a human index/README exists, check it agrees with the manifest — same set of in-scope concerns and the same `current_rung` per concern. A README that lists a concern the manifest omits (or shows a different rung) is **README↔manifest drift** → report as a warning (the manifest is authoritative; the index is derived and should be regenerated). Don't audit the README's prose, only this agreement.
9. **Status↔claim consistency (Part 6).** The publish state must match the claims. If the manifest/README claims build-readiness (every in-scope concern at its target rung), **or** the set is demonstrably built (a populated binding map / a build-status record), then every in-scope doc must be `status: published` with **no build markers** (`<!-- BUILD:`). Report each violating doc as a **publish-step-owed** finding — an **error** when build-readiness is claimed or a build exists, a **warning** when the set is merely at target with no claim yet. The reverse — a `published` doc still carrying build markers — is always an **error** (a half-run publish step). A `draft` doc whose delta is still being authored (not yet handed off) is work in progress, not a finding. Note the framing: this check is **evidence-based detection** — implementation artifacts *prove* an authoring gate was skipped — not a coupling of doc status to implementation; publish itself transitions on authoring events only (Part 6).
10. **Build-readiness:** report whether every in-scope concern has reached its target rung **and the set is published (Part 6)** — together, the build-ready gate — and list the blockers if not. **The binding map is not part of this gate** — it is authored *during* the build (the realizing slice records each binding), so its absence pre-implementation is expected, not a finding.

## Output
A concise report: per-concern (actual rung / claimed rung / gaps), vocabulary re-partition findings (unaccounted keys, with the version delta when attributable), dangling references, scope warnings, publish-state findings, and an overall build-ready verdict. No edits, no file changes.

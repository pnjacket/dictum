# Concern 11.2 — Domain & Data

> **One-line:** What the data *is*, what rules it must obey, and how it persists. Owns the canonical entity definitions the rest of the set is shaped against.

## Purpose & Scope

Defines the conceptual **domain model** (entities, relationships, invariants) as the canonical truth, and the **storage model** that realizes it. Owns the `ENTITY-###` definitions and field-level classification tags. It does **not** own protection/retention *policies* (Security & Governance) nor the wire shapes that project from entities (Interfaces).

## Owned Contracts (referenced by stable ID)

1. **Entity definitions** — `ENTITY-###`: fields, types, identifiers, relationships. **Referenced by** Interfaces (DTOs/wire shapes project from these), User Experience (what's displayed/edited), Security & Privacy + Governance (act on classification tags), Quality & Testing (test data shape). Where user accounts / identity exist in the product, includes the **user entity** `ENTITY-USER`, linked to `PERSONA-###` (Product & Requirements); a product with no accounts (e.g. a client-only browser game) has no `ENTITY-USER`, and its absence needs no Non-goals defense.
2. **Invariants** — `INV-###`: business rules stated as checkable conditions. **Referenced by** Quality & Testing (each becomes assertable).
3. **Classification tags** — field-level `pii` / `sensitive` / `secret` flags. **Referenced by** Security (protection policy) and Governance (retention/residency policy).

## Consumed (referenced, not owned)

- `PERSONA-###` (Product & Requirements) — `ENTITY-USER` links to personas.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Domain & Data |
|---|---|
| **Absent** | No data model. |
| **Sketch** | Key entities and relationships as bullets. |
| **Specified** | Entities described with fields, relationships, and lifecycle in prose; invariants stated narratively; storage approach noted. |
| **Contract-grade** | `ENTITY-###` with typed fields + identifiers (**id format specified, incl. UUID version where it carries semantics**); relationships explicit; invariants as checkable `INV-###` (**a uniqueness / exactly-once `INV-###` names its enforcement mechanism**; an `INV-###` that depends on a derived key specifies the derivation); lifecycle states defined; sensitive fields tagged; storage schema specified (when persistence is in-scope). An implementer can build the model and a test can assert its invariants without guessing. |
| **Verified** | *(delivery methodology)* The built schema enforces the invariants; tests assert them. |

## Contract-grade bar

Entities + types + stable IDs + relationships + checkable `INV-###` invariants + lifecycle states + classification tags (and storage schema if persistence is in-scope). Three additional bars on identifiers and invariants:

- **Enforced ≠ checkable.** An `INV-###` that asserts **uniqueness or "exactly once"** must **name its enforcement mechanism** — a DB constraint, a unique index, a transaction, or a **structural / by-construction** mechanism in stores that have no declarative constraints (a fixed primary key, a single-record `put()`, a transaction scope): what matters is that the store *cannot represent* the violation, not which feature prevents it. An invariant enforced only by a test — not by the schema — can still be violated under concurrency and must be explicitly marked advisory.
- **An identifier's format is part of its contract.** State the id format precisely, **including the UUID *version* where the version carries semantics** (e.g. a time-ordered v7 id underpinning an ordering `INV-###`). The version is owned here; downstream validators (Interfaces 11.4) must honor it.
- **A derived key is itself a contract.** When an `INV-###` depends on a **derived key** (e.g. a canonical key computed from a sorted member set), specify the derivation **once** — the inputs, ordering, encoding, and delimiter — so independent modules computing it converge.

# Concern 11.9 — User Experience

> **One-line:** The screens, the navigation that connects them, the journeys through them, and the states they show — each screen wired to the capability it realizes, the interface it calls, and the role that guards it.

## Purpose & Scope

Active when the product has an interactive UI. Owns the **navigation contract** and **screen specifications**, binding each screen to its capability, interface, and guard. It does **not** own a11y/i18n (Concern 11.14, referenced) nor the design system itself (design-tool binding `[REVISIT]` — blocked-by-environment).

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Default (when UI present) | Notes |
|---|---|---|---|
| Navigation / IA contract | `navigation-ia-contract` | in | `SCREEN-###` (+ `ROUTE-###` **where a URL surface exists** — see below) + menu + guard + reachability |
| Screen specifications | `screen-specifications` | in | realizes `CAP`, calls `API`, gated by `ROLE` |
| States | `states` | in | **display** states (loading / empty / error, not just happy path) **and interaction/input states** — each interactive control's default/empty submission (a blank/“all” filter, an unset field) is a spec'd state. **For a live/event-driven UI, also the event-driven / ambient states** (see below) |
| User journeys | `user-journeys` | in | what the real-flow E2E traverses |
| Component reuse / design-system binding | `component-reuse-design-system-binding` | in when a component library exists | reuse shared library — **this IS the rule** while no design tool is in play; design-tool binding is `[REVISIT]` (blocked-by-environment). **Defaults out when no library exists** (e.g. a zero-dependency hand-rolled UI) — what remains is the in-product visual-consistency rule, owned by the screen specs |
| Responsive / layout | `responsive-layout` | optional | |

### Event-driven / ambient states (live UIs)

The display/interaction/input states above are **pull-driven**. The ambient bar triggers on **any state change not produced by a user action** — a server-pushed event surface (Interfaces `EVT-###`) *and* **client-local asynchrony** (a timer firing an autosave, a pointer-lock loss pausing the app, async worker results, completion of a background load). The Contract-grade bar **additionally** covers such states per screen, or per the persistent shell where they live globally — enumerated as an **ambient-state register** when shell-global. For the server-pushed instance the canonical set is:

- **Connection state** — `connected` / `reconnecting` / `disconnected` / `backfill` (catching up after a reconnect). The UI must show which it is and behave sanely in each (e.g. queued vs dropped actions while disconnected).
- **Presence** — who is online/away/offline, where presence is surfaced.
- **Typing** — typing indicators (a high-churn ephemeral state).
- **Unread dividers / counts** — the "new since you left" boundary and unread markers, and how they resolve on read.
- **Server-pushed error frames** — a server-pushed error event (`EVT-S-ERROR`) is a first-class state routed through the same total, content-negotiated error model (Interfaces 11.4 / Architecture 11.3), not a raw frame.

### SPA-shell pattern (one persistent shell, layered screens)

A `SCREEN-###` need not be a distinct page. The **SPA-shell pattern** is blessed: one persistent shell (e.g. a 3-pane layout, where connection/presence/ambient states typically live) with most screens realized as **panels or modals layered over it**. Each such screen is still a `SCREEN-###` with the full contract, still **deep-linkable** and **reachability-checked** — no-orphans is judged by "every screen is reachable and every link resolves" exactly as for page-like screens.

`ROUTE-###` ↔ `SCREEN-###` is **N:1** (a screen may be reachable by several routes — a canonical path plus query-param / modal routes) and never N-screens-per-route. For a client-routed SPA, `ROUTE-###` names client routes (including query-param and modal routes), not only server paths; each `ROUTE-###` resolves to exactly one `SCREEN-###`.

### Routeless products (no URL surface)

`ROUTE-###` is **conditional on a URL surface existing**. A product whose screens are app states rather than URLs (e.g. a pointer-locked canvas game with title/playing/paused states) mints **no** `ROUTE-###` (that would be the spurious-prefix anti-pattern); instead the navigation contract binds to an **owned state-machine register** (`NAV-###` transitions over the app-state set). No-orphans generalizes: **every screen is reachable via a spec'd transition, and every transition resolves to an existing screen**. Deep-linkability applies only where URLs exist.

### The `layer` screen kind

Some surfaces are never *navigated to*: a debug overlay, a toast, a banner. These are a third screen kind — a **layer** — alongside page-like screens and SPA-shell panels/modals. A layer carries the screen contract like any other, but its reachability fact is its **toggle/trigger** (the input that shows it, or the ambient event that raises it); a purely ambient layer may equivalently be specified as rows of the ambient-state register above. An unclassified overlay is an orphan by another name.

### Modal integrity

**A modal's inertness over the background must be input-channel-complete: pointer AND keyboard.** A scrim that intercepts clicks while Tab/focus traversal still reaches background controls is **not a modal** — background interaction through *any* input channel is a correctness bug in this concern's bar, not an accessibility refinement. Accessibility (11.14) owns only the focus-order *refinement* within an already-inert modal (initial placement, wrap order, restore-on-close); the inertness itself is owned here.

## Owned Contracts (referenced by stable ID)

1. **Navigation contract** — `SCREEN-### / ROUTE-###` (or `SCREEN-###` + the state-machine register, for routeless products): each entry = route/spec'd transition + menu placement + guard — **the `ROLE-###` where an auth surface exists, else the screen's enabling condition** (e.g. "a valid save exists") — + reachable-from parent + realized `CAP-###` + **the interface surface it drives** (the called `API-###` where one exists; the `LIB-###`/input-binding surface for a client-only product) + required states + **its interactive controls** (each form/button + the interface element it submits to or drives + its default/empty-input state). **Referenced by** Delivery Process (the nav-wiring DoD item; slices), Quality & Testing (the real-flow E2E **operates** these controls — 11.5), Security (guards).
2. **User journeys** — `JOURNEY-###`: the real path a `PERSONA-###` takes. **Referenced by** Quality's real-flow E2E.

## Consumed (referenced, not owned)

- `CAP-###`, `PERSONA-###` (Product & Requirements); `API-###` (Interfaces); `ROLE-###` (Security); `ENTITY-###` (Domain, for display).
- Accessibility & Internationalization (11.14); the design system (design-tool binding `[REVISIT]`, blocked-by-environment).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for User Experience |
|---|---|
| **Absent** | No UI (a non-UI product — concern not in scope). |
| **Sketch** | Screen list + rough navigation, as bullets. |
| **Specified** | Screens described with content; journeys outlined; navigation in prose; states noted. |
| **Contract-grade** | `SCREEN-###` with route (or spec'd transition) + nav-wiring + guard (role or enabling condition) + realized `CAP-###` + the interface surface it drives + required states (loading/empty/error) + **enumerated interactive controls and their default/empty-input states** + **event-driven/ambient states for any non-user-action state change** (server push or client-local) + **input-channel-complete modal inertness** + journey placement; component-reuse rule applied where a library exists. An implementer builds each screen wired into real nav and its real interface surface, with all states and controls, no guessing. |
| **Verified** | *(delivery methodology)* Screens built, wired, and the real-flow E2E passes. |

## Contract-grade bar

Per screen: route (or spec'd transition where no URL surface exists) + nav-wiring + guard (`ROLE-###` where auth exists, else the enabling condition) + realized `CAP-###` + the interface surface it drives (`API-###` where one exists) + required states + **enumerated interactive controls with their default/empty-input states** + journey placement; component-reuse rule applied where a component library exists (the rule while no design tool is in play).

The subsection rules apply as part of this bar: **event-driven/ambient states** for any state change not produced by a user action; the **SPA-shell** and **layer** treatments (a `SCREEN-###` may be a panel/modal over a persistent shell — still deep-linkable, reachability-checked, `N:1` with its `ROUTE-###` — or a layer whose reachability fact is its toggle/trigger); and **input-channel-complete modal inertness** (pointer AND keyboard).

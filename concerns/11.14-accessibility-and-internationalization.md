# Concern 11.14 — Accessibility & Internationalization

> **One-line:** Whether everyone can use it (a11y) and whether it works across languages and regions (i18n) — two independent sub-aspects that switch on separately.

## Purpose & Scope

**a11y** activates with any human-facing UI; **i18n** activates with multi-locale scope — either can be on alone. First-class concern that binds its requirements to UX screens and locale-affected entities. Tested via Quality's a11y/i18n test types.

### Accessible-announcement contract (live-updating UIs)

The facet list above (keyboard / contrast / semantics / focus) is **static-DOM** a11y. A product with a **live-updating UI** — content that changes without a user action (a server-pushed message arriving, a status changing) — has an obligation those facets never cover: a screen-reader user must learn the content updated **without losing their place**.

For any live-updating UI, the a11y Contract-grade bar **additionally** requires an **accessible-announcement contract** for asynchronous content updates:

- **An ARIA live region** announces incoming updates, with a **stated politeness** (`polite` vs `assertive`) per kind of update.
- **Without stealing focus** — the announcement reaches assistive tech but the user's caret/focus stays where it was.
- **Burst throttling / coalescing** — under a burst (many messages at once) announcements are throttled/coalesced rather than read one-by-one.
- **Deliberate exclusions** — high-churn ephemeral events are **explicitly excluded** from announcement; the contract names what is silent and why. The exclusion kind is anything that updates constantly *by design* and would bury real announcements — typing/presence indicators, and equally non-messaging churn (frame/telemetry readouts, periodic timer events, background content streaming).
- **Ownership split with the node's owner** — the concern that owns a DOM node fixes the *node* (structure, roles, names); **whether content inserted into a live region is announcement-eligible belongs to *this* contract**, keyed to the update's trigger/provenance (a user-initiated save announces; a timer-driven one may be deliberately silent). The node owner must not pre-decide eligibility by where it places content.

## Owned Contracts (referenced by stable ID)

a11y contracts use the **`A11Y-*`** ID family. Each is **owned per `SCREEN-###`** (or per the persistent SPA-shell where a contract is global, e.g. the live-region announcement contract): a per-screen `A11Y-###` carries that screen's conformance specifics, and where it matters the announcement contract.

1. **a11y requirements per `SCREEN-###`** (`A11Y-###`) — conformance level + per-screen specifics + (live UI) the accessible-announcement contract.
2. **i18n requirements** — per `SCREEN-###` and per locale-affected `ENTITY-###`.

## Consumed (referenced, not owned)

- `SCREEN-###` (User Experience); locale-affected `ENTITY-###` (Domain); Quality's a11y/i18n test types.

## Contract-grade bar

Conformance target per screen (`A11Y-###`) + **(if the UI is live-updating) the accessible-announcement contract** (stated above) + (if i18n on) externalized strings + locale formats + RTL + pluralization + translation process.

- **A conformance claim = level X minus traced SC exceptions.** Each excepted success criterion is named and traced to the scope-out that creates it — and that scope-out may legitimately live in *another* concern (e.g. UX scoping out `responsive-layout` punches a justified hole in a reflow SC). A **traced** cross-concern hole is legal; an untraced silent shortfall is what fails the bar. The claim's value is that its exceptions are enumerable, not that it is exceptionless.
- **A focus-placement rule states *when* it binds.** Where the contracted landing element depends on an **asynchronously-resolved enabling state** ("the primary button *if enabled*, else the fallback"), the contract says whether the landing is asserted **at state entry** (per the then-current DOM) or **after the enabling read settles** — and a late re-application may only move focus that still sits where arrival placement left it, never focus the user has since moved. Silent on this, the contracted element is unknowable at placement time and the test races the read.
- **A recommended mechanism is checked against sibling concerns' test surfaces.** A mechanism this concern *recommends* (e.g. `inert` for modal background containment) can collide with another concern's E2E-verified contract on the same DOM (e.g. a UX capture-at-interaction behavior exercised by editing a background field). The contract here is the **observable** ("Tab never reaches outside the open layer"); the mechanism stays delegated, and a recommendation that breaks a sibling's verified surface yields to any mechanism satisfying the observable.

# Concern 11.6 — Delivery Process

> **One-line:** How work is sliced, what "done" means, in what order it's built, and how a build-ready spec becomes a verified product. The bridge from the doc system to the running software.

## Purpose & Scope

Owns the **process spine**: the vertical-slice rule, the Definition of Done, the work-item hierarchy and slice-level designation, the build playbook, the build-ready gate scope, and **implementation-status (Verified) tracking**. It owns the `Verified` rung that every other concern hands off to. The slice/DoD/playbook rules are a **near-universal reusable core** (shipped as a template); only the hierarchy mapping, slice list, and gate scope are per-product.

## Owned Contracts (referenced by stable ID)

1. **Slice rule + fidelity-staged Definition of Done** — the reusable template (≈ `docs/00-delivery-methodology.md`). The DoD has **two gates** referencing Operations' environment fidelity map: a **merge gate** (developer stage — own code real, external deps substitutable per the map; the bar for code entering the repo) and a **release gate** (real-infra, owned by DevOps). Real-infra checks not runnable at merge are deferred to the release gate and **tracked, never silently skipped**. For a product with **no substitutable externals**, the substitution half is **stated as empty, not omitted** (an explicit "no externals; nothing substituted") so the gate stays checkable rather than reading as if something must be substituted.

   **Slice ≠ a `CAP-###` bijection.** A slice is a vertical cut that ends in a verified increment, **not** a per-`CAP-###` row. A **cross-cutting / infrastructure slice** is first-class: a real `slice:full` that lights up a **shared path** used by several capabilities, verified end-to-end on its own, with the per-capability slices that ride that path landing after it; it maps to *several* `CAP-###` (or to none directly, when it's pure shared infrastructure). A **verification-only slice** — one whose increment is *proof* (a test suite, an audit pass, a hardening battery) over an already-built surface — is likewise first-class. Do **not** force a strict one-slice-per-`CAP-###` bijection; do **not** invent a `SLICE-###` ID web (slices are process artifacts, not ID-prefixed contracts). The playbook's "no deferred frontend/E2E" rule is read at **capability granularity**: no `CAP-###` completes without its real-flow E2E, but headless foundation slices (`slice:headless` layers a later journey rides on) may legitimately precede the first UI journey.
2. **Work-item hierarchy declaration** — which level is the slice level in this product's tracker (≈ `docs/00a-work-item-hierarchy.md`).
3. **Build-ready gate scope** — the set of in-scope `CAP-###` / concerns / sub-aspects whose Contract-grade triggers "build-ready".
4. **Build-status record** — per `CAP-###`/slice: built & verified, and **at which fidelity stage** (merge / release). **Kept separate from the manifest's documentation rung** (doc maturity ≠ implementation status).
5. **Tracker-binding declaration (optional)** — present only when work items live in an external tracker. A thin, product-local **adapter** (implementing the *execution-mirror* boundary) declaring **(a)** the *ID-carrying channel* — where a repo contract ID rides on a tracker item (a field / label / link) so the item *references* the contract, never restates it; **(b)** the *projection direction* — the derived backlog (Std Part 10e) and the build-status record drive item creation + status **repo→tracker only**; **(c)** the *reconciliation rule* — **repo-wins** (a divergent tracker edit is a convenience, not a source). **Three item roles** (transitions of one item, not separate objects — the role is fixed by whether the item yet references a Contract-grade repo ID): **demand** — a request for new/changed behavior, ingested via `doc-feature`; **triage** — an unadjudicated defect hypothesis (repo-invisible until reproduced; then an externally-sourced `drift`-type change event, Std Part 10d); **execution** — a slice referencing contract IDs (the mirrored, buildable role). **Two gates:** *build* — no item is built until it references a repo contract ID at Contract-grade; *write* — a suspected defect earns its first repo write only on reproduction + adjudication (a code-defect verdict revokes `Verified` via a regression case and leaves the contract text untouched). **Boundary invariant:** the repo owns contracts and `Verified`/build-status; the tracker mirrors those (repo→tracker) and may additionally hold upstream demand and triage hypotheses. The repo **never** stores a tracker item ID/URL (Std Part 0.5); the only join is the stable ID, resolved at projection time.

## Consumed (referenced, not owned)

- `E2E-STANDARD` (Quality & Testing) — the proof-of-done gate references it.
- Environment **fidelity map** (Operations & Infrastructure) — the staged DoD's merge/release gates reference it.
- `API-###` etc. (Interfaces) — a slice's work item references the contracts it touches.
- `CAP-###` (slices realize capabilities), `COMPONENT-###` (slices map to components).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Delivery Process |
|---|---|
| **Absent** | No defined process. |
| **Sketch** | "We'll build in slices with tests", as bullets. |
| **Specified** | Slice rule, DoD, and sequence described in prose; hierarchy noted. |
| **Contract-grade** | Slice rule + full DoD + declared hierarchy/slice-level + build playbook + build-ready gate scope + Verified-tracking method, all concrete. An implementer can run the build with no process guesswork. |
| **Verified** | *Reflexive:* the process was actually followed and produced a verified product. (This concern *defines* Verified for all others.) |

## Contract-grade bar

Slice rule + full DoD + declared hierarchy/slice-level + build playbook + build-ready gate scope + Verified tracking — all concrete; and, when an external tracker is used, the tracker-binding declaration (channel + roles + gates).

# Concern 11.11 — Observability & Monitoring

> **One-line:** What the running system reveals about itself — logs, metrics, traces, SLOs, alerts — and the audit trail that realizes Governance's audit policy.

## Purpose & Scope

Active when the product is a running service. Owns **telemetry** and the **audit-trail mechanism**. It does **not** own performance *targets* (Performance & Scalability) nor audit *policy* (Governance) — it implements/measures both.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Default (when deployed) | Notes |
|---|---|---|---|
| Per-`COMPONENT` telemetry | `per-component-telemetry` | in | Logs / metrics / traces emitted |
| SLIs / SLOs | `slis-slos` | in when targets exist | Measure Performance's targets |
| Alerting rules | `alerting-rules` | in for production | Each rule guards an `SLO-###` and routes to a role + runbook |
| Audit-trail mechanism | `audit-trail-mechanism` | in when audit required | Realizes Governance `POLICY` |
| Dashboards | `dashboards` | optional | |
| Log/audit retention | `log-audit-retention` | in when regulated | Per Governance policy |

## Owned Contracts (referenced by stable ID)

1. **Per-`COMPONENT` telemetry spec** — logs/metrics/traces each `COMPONENT-###` emits.
2. **SLO/SLI definitions + alert rules** — `SLO-###`, measuring Performance's targets, and `ALERT-###` (a local prefix) for the rules that guard them. **Referenced by** Operations (runbooks/alerts).
3. **Audit-trail mechanism** — realizes Governance's audit `POLICY-###` (structured, tamper-evident where required) with stated retention.

## Consumed (referenced, not owned)

- `COMPONENT-###` (Architecture); audit `POLICY-###` (Governance); performance targets (Performance & Scalability).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Observability & Monitoring |
|---|---|
| **Absent** | Not deployed / no telemetry. |
| **Sketch** | What's logged/monitored, as bullets. |
| **Specified** | Telemetry approach per signal; alerts noted. |
| **Contract-grade** | Per-`COMPONENT-###` signals + `SLO-###`/SLIs + alert rules + audit trail per Governance policy + retention. An operator can observe, alert, and audit without guessing. |
| **Verified** | *(delivery methodology)* Telemetry emitting, alerts firing, audit captured. |

## Contract-grade bar

Per-component signals + SLOs/SLIs + alert rules + audit trail per Governance policy + retention.

- **An alert rule is a complete routing row, not a bullet.** When `alerting-rules` is in scope, each rule states: the `SLO-###` it guards, its trigger (threshold + evaluation window, or an error-budget burn rate), its severity, a **`ROLE-###` notification target — a role, never a person** — and its `RUNBOOK-###` response reference, so the row is wireable into any alerting backend as written. An unguarded SLO (no alert row) is a deliberate scope-out, recorded in *Non-goals*, not an omission.

# Concern 11.4 — Interfaces & Contracts

> **One-line:** Every surface through which the product is invoked, specified as a contract precise enough that both sides build to it without guessing. The single source the delivery methodology's work items reference.

## Purpose & Scope

Owns the canonical contracts for each active interface kind. One concern, **sub-typed by trait**: a product activates the sub-types it exposes (HTTP API, CLI, library/SDK, events, UI entrypoints) — often several at once. This is the primary anti-drift mechanism in the whole system.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Activated when | Notes |
|---|---|---|---|
| HTTP/RPC API surface | `http-rpc-api-surface` | exposes a network API | `API-###` |
| CLI surface | `cli-surface` | has a command-line interface | `CLI-###` (commands, flags, exit codes) |
| Library / SDK surface | `library-sdk-surface` | consumed as a library | `LIB-###` (public functions/types) |
| Event surface | `event-surface` | publishes/consumes events | `EVT-###` (event schemas, topics). **Intra-process/worker message protocols** (e.g. worker `postMessage` payloads) are wire schemas too — suggested prefix `MSG-###`, on the same footing as `EVT-###` |
| UI entrypoints | `ui-entrypoints` | has a UI | The **navigation contract** — owned by User Experience (11.9), referenced here. Interfaces still owns the contract layer **below** screens: input bindings (key/pointer → action) and the stable DOM hooks E2E/a11y tests select against — suggested id schemes `BIND-###` / `DOM-###` |
| Error model / catalog | `error-model-catalog` | always (for in-scope kinds) | Shared `ERR-###` envelope + per-element errors |
| Versioning & compatibility | `versioning-compatibility` | scales with exposure | Detailed for public/library; light/out for internal & PoC |
| Pagination / filtering / rate-limit conventions | `pagination-filtering-rate-limit-conventions` | optional (APIs) | May reference Performance / Ops |

## Owned Contracts (referenced by stable ID)

Each programmatic element (`API-### / CLI-### / LIB-### / EVT-### / MSG-###`) carries:
`signature · typed inputs/outputs (ref ENTITY-###) · error catalog · pre/postconditions · side-effects/idempotency · owning COMPONENT-### · served CAP-###`.

The following rules apply to every in-scope element — the request/response surface **and** the `event-surface` (`EVT-###`, and `MSG-###` intra-process/worker message protocols) on the same footing:

- **Concrete response projection, not just the entity.** The typed output must name the wire shape — the fields exposed and their wire names (the DTO a client builds against) — not merely the owning `ENTITY-###`. "Returns `ENTITY-X`" is not build-ready: it leaves field selection and casing (snake_case vs camelCase, which fields withheld) for each layer to invent.
- **Emitted documents are contracts of their own (`OUT-###`).** Where an element's load-bearing output is a structured document — a JSON report, CSV export, rendered Markdown summary — rather than a projection of a persisted entity, its shape is owned as an **output-document schema** `OUT-###`: top-level structure, field/column/section names + types + units, ordering, and which parts consumers may rely on across versions. The emitting element references it as its typed output, and the response-projection rule applies unchanged. An `OUT-###` needs no sub-aspect key of its own — it rides whichever in-scope surface emits the document.
- **Inputs specify optionality + empty-value semantics.** For each parameter: required/optional, and what an empty value means (e.g. a blank/“all” filter = no filter, *not* a parse error). The value a UI control submits when unset (typically `""`) is part of the contract.
- **The error model is total and content-negotiated.** *Every* error — including errors raised by the framework/runtime (input validation/parsing), not only those the application raises — maps to the shared `ERR-###` envelope and is **content-negotiated**: an HTML client receives the HTML error rendering; a programmatic client receives the structured error. A raw framework error reaching the caller is a contract violation (a contract test asserts it). *Client-only adaptation:* where no programmatic caller exists (a single in-browser surface), the negotiation half degenerates and the operative rule is **total + never console-only** — every error reaches a user-visible surface, never only the developer console (cross-ref Architecture 11.3).
- **Event/subscription authorization.** An event element specifies its payload projection, its error/failure behavior, and its **subscription/connection authorization** (handshake auth + per-resource subscribe checks; cross-ref Security 11.7) on the same footing as an `API-###`.

**Referenced by** Delivery Process (a slice's work item links the IDs it touches), User Experience (which calls which `API-###`), Quality & Testing (contract/integration tests), Integrations (external-facing contracts).

## Consumed (referenced, not owned)

- `ENTITY-###` (Domain & Data) — I/O shapes project from entities.
- `COMPONENT-###` (Architecture) — each element is owned by a component.
- `CAP-###` (Product & Requirements) — each element serves a capability.
- Security & Privacy — the authz applied to each element.
- UX navigation contract — for UI route entrypoints.

## Contract-grade bar

Every in-scope interface element fully specified per Owned Contracts (signature + typed I/O + parameter optionality/empty-value semantics + error catalog + pre/postconditions + side-effects + owning component + served capability), with a **concrete response projection** and, where the output is a structured document, an owned **`OUT-###`** schema; the `ERR-###` model **total** (covers framework/validation errors) and **content-negotiated**; **event-surface parity** for any in-scope `EVT-###`/`MSG-###`, including subscription authz; versioning detailed where the interface is exposed. Both sides can build, and a contract test can call it, without guessing. Three further requirements:

- **Validators honor the owned id format.** Input validation for an identifier must match the format Domain 11.2 owns, **including the UUID version** where it carries semantics (a v4-default validator rejecting v7 ids is a contract violation).
- **One canonical bound.** A parameter/collection bound is stated on **one canonical quantity** and the other derived (e.g. state the size limit on *total* members, derive the *others* bound), so the DTO bound and the owning `INV-###` (Domain 11.2) cannot disagree.
- **A forced error names a forcing the proving tier can use.** Where an error-catalog row records how its condition is forced under test (a *Forced-by* cell) **and** is assigned a verification tier, the two are **authored together**: the named forcing mechanism must be usable at the tier that proves the row (a row owed an end-to-end "forced condition → user-visible surface" proof cannot list only unit-level forcings such as stubs and fake timers).

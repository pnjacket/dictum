# Concern 11.3 — Architecture

> **One-line:** The logical structure of the system — what the components are, what they're responsible for, how they interact, and the patterns and decisions that hold it together.

## Purpose & Scope

Defines the **logical** design: component decomposition, responsibilities, boundaries, interactions, cross-cutting patterns, technology choices, and the cross-cutting decision record. It does **not** own **physical** deployment/runtime (Operations & Infrastructure realizes the component model).

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Typical default | Notes |
|---|---|---|---|
| Component decomposition & responsibilities | `component-decomposition-responsibilities` | always in | The structural map. Includes **in-process execution placement** (e.g. worker vs main thread) where it is load-bearing |
| Boundaries & isolation model | `boundaries-isolation-model` | in for multi-component systems | Trust/data/process boundaries |
| Component interactions / data flow | `component-interactions-data-flow` | in for multi-component systems | Sync/async, who calls whom |
| Cross-cutting patterns | `cross-cutting-patterns` | in for non-trivial systems | Auth, error handling, logging, config |
| Technology choices | `technology-choices` | always in | With rationale (→ ADRs) |
| ADR register | `adr-register` | always in | Cross-cutting decisions (`ADR-###`) |
| Logical deployment topology | `logical-deployment-topology` | in for distributed systems | *Logical* only; physical → Ops. **In-process** concurrency placement is *not* topology — it belongs to component decomposition/interactions, so scoping this key out does not scope that out |
| Scalability / resilience patterns | `scalability-resilience-patterns` | optional | Overlaps Performance & Scalability |

## Owned Contracts (referenced by stable ID)

1. **Component model** — `COMPONENT-###`: responsibility, owned interfaces, dependencies. **Referenced by** Interfaces (each endpoint/command attributed to a component), Delivery Process (slices map to components), Operations (deploy units), Observability (telemetry per component).
2. **ADR register** — `ADR-###`: system-wide/cross-cutting decisions with context · status · consequences. **Referenced by** any concern (and the failure-mode rationale). Concern-*local* decisions stay in that doc's Design Decisions section.
3. **Cross-cutting patterns** — `PATTERN-###` (e.g. auth propagation, error envelope, structured logging). **Referenced by** every implementing concern so behavior is uniform. A cross-cutting **error-handling `PATTERN` must be *total*** — handling errors from *every* source, including the framework/runtime (input validation/parsing), not only application-raised ones — **and content-negotiated** (HTML vs programmatic clients); for a **client-only** product with a single presentation surface the negotiation half degenerates to **total + never console-only** (every error reaches a user-visible surface, not just the developer console). This realizes the Interfaces (11.4) total error-model rule, and closes the leak where a framework's *default* error response bypasses the uniform envelope.

## Consumed (referenced, not owned)

- `CAP-###` (Product & Requirements) — components realize capabilities.
- `ENTITY-###` (Domain & Data) — components own data.
- Physical realization (Operations & Infrastructure).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Architecture |
|---|---|
| **Absent** | No architecture doc (acceptable only for the most trivial single-file tool). |
| **Sketch** | High-level boxes — main components and how they connect, as bullets/a diagram. |
| **Specified** | Components described with responsibilities; boundaries and interactions in prose; tech stack chosen; key patterns named. |
| **Contract-grade** | `COMPONENT-###` with responsibility + owned interfaces + dependencies; cross-cutting `PATTERN-###` specified concretely enough to implement uniformly; `ADR-###` for key choices; boundaries explicit. An implementer can place new code correctly and follow consistent patterns without guessing. |
| **Verified** | *(delivery methodology)* The built system matches the component model and patterns (e.g. via structure/fitness checks). |

## Contract-grade bar

Components (responsibility + owned interfaces + dependencies) + specified cross-cutting patterns (incl. the total error-handling `PATTERN` per Owned Contracts) + ADRs for key decisions + explicit boundaries.

# Concern 11.10 — Operations & Infrastructure

> **One-line:** How the system is physically deployed, configured, provisioned, run, and recovered — and the environment fidelity map that says what's real vs substituted at each stage.

## Purpose & Scope

Active when the product is deployed as a running service — **or, more generally, for any product whose staged DoD needs an environment fidelity map** (Part 10a requires one). A client-only product with no server still has owned environment truth: the browsers/runtimes it must run on, the fidelity at which tests run against them, and the serving contract for its static artifact — so the trigger is "an `ENV-###` map is needed", not "a service runs". Owns the **physical realization** of the `COMPONENT-###` model and the **environment fidelity map**. It does **not** own telemetry (Observability & Monitoring).

## Owned Contracts (referenced by stable ID)

1. **Environment fidelity map** — `ENV-###`: each environment declares, per external dependency, real vs approved-substitute, and an explicit **`auth:`** assertion as a checkable per-environment field, not prose — `real, no bypass` where an auth surface exists, or the enumerated value **`n/a — no auth surface exists`** for a no-auth product (the field stays checkable either way; it is never simply absent). Fidelity **may differ by test tier within one environment** (e.g. a real S3 container at the gate's E2E but a Mock-S3 in unit tests) — when it does, the **gate's E2E fidelity is what binds**. Alongside the per-external column, the map names the **platform/host fidelity dimension**: the *runtime itself* can be the substituted thing (a software-GL renderer standing in for hardware GPU, a headless engine for a headed display) — and the **browser/runtime build is itself a fidelity fact** to record (a harness-bundled or patched engine is a *proxy* for the branded browser the contract names). **Referenced by** Quality & Testing (maps test stages) and Delivery Process (the staged DoD's merge/release gates). The single source of environment truth.
2. **Deployment spec per `COMPONENT-###`** — build, deploy unit, scaling, resources, runtime config. Architecture owns logical; Ops owns physical; `COMPONENT-###` is the join.
3. **Runbooks** — `RUNBOOK-###`: deploy, rollback, recovery, incident.
4. **Dev-toolchain register** — `TOOL-###`: the version-pinned **build/test-time-only tooling** (compiler/type-checker, test runner, static dev server, automation harness). Such tooling is **not** a `DEP-###` (11.12 owns *runtime* externals); its register lives **here**, co-owned with Quality & Testing for the tiers that run it — including (especially) when 11.12 is out of scope entirely. **Pinning `TOOL-###` is itself an environment fact, so a product with a toolchain to pin *needs* an `ENV-###` map and thereby activates this concern (the broadened trigger above) — the register always has a single home, in Ops. A set must not scope Operations out while pinning build/test tooling; if it has tooling to register, Ops is in.**
5. **Committed-config-key register** — `CONFIG-###`: each **committed, behavior-shaping** configuration key, as a contract: name · type/units · default · the invariant/behavior it drives, **by ID reference** (the `INV-###`/`CAP-###` a retention cutoff or a cleanup cadence feeds). The distinction from `ENV-###` is crisp: **`ENV-###` records environment/infrastructure facts** — what runs where, what is real vs substituted; **`CONFIG-###` records a product-behavior parameter the code reads.** An environment may *set* a key's value (per-`ENV` overrides belong in the deployment spec / fidelity map rows), but the key's contract — type, default, driven behavior — is owned once, here. A key that shapes no contracted behavior (a log level, a bind port) stays plain prose in the runtime-config surface; `CONFIG-###` is for the keys a contracted behavior turns on.

## Consumed (referenced, not owned)

- `COMPONENT-###` (Architecture) — the logical units being realized.
- Integrations & External Dependencies (11.12) — the externals the fidelity map substitutes.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Operations & Infrastructure |
|---|---|
| **Absent** | Not deployed (a library/CLI distributed as an artifact — concern not in scope). |
| **Sketch** | How and where it runs, as bullets. |
| **Specified** | Environments described; deployment approach; config noted. |
| **Contract-grade** | `ENV-###` + fidelity map + per-`COMPONENT-###` deployment spec + runtime config (behavior-shaping keys minted as `CONFIG-###`) + reproducible provisioning + key runbooks. An operator can deploy, scale, and recover without guessing. |
| **Verified** | *(delivery methodology / DevOps release gate)* Deployed and recoverable; provisioning reproduces the environment. |

## Contract-grade bar

Environments + fidelity map + per-`COMPONENT` deployment spec + runtime config + reproducible provisioning + key runbooks (deploy/rollback/recovery). An operator can deploy, scale, and recover without guessing.

The `ENV-###` rules — fidelity may differ by test tier (the gate's E2E fidelity binds the staged DoD), the checkable `auth:` field with its enumerated no-auth value, and the platform/host fidelity dimension — and the `CONFIG-###` rule (behavior-shaping committed keys are minted as contracts, not left as prose in an `ENV-###` row) are stated once under Owned Contracts and bind at this rung. Beyond those:

- **Multi-store backup unit = the consistent set.** When durable state spans multiple stores, the backup/restore unit is the **consistent set** (e.g. the database dump paired with the object-store snapshot), not any single store; a backup of one store alone is not Contract-grade.
- **Real-time/WS products state an ingress/edge concern.** Protocol upgrade and ingress routing are explicit; **session affinity is not required when fan-out is external** (a shared pub/sub broker delivers across nodes), and the map should say so rather than leave affinity implied.

# Concern 11.13 — Performance & Scalability

> **One-line:** How fast, under how much load, and how it grows — the measurable targets, the load it must hold, and the strategy for scaling to meet them.

## Purpose & Scope

Active when there are real throughput/latency/scale requirements. Owns performance **targets**, **load profiles**, and the **scaling strategy + capacity model**. Architecture provides enabling patterns; Operations implements the scaling mechanism; Observability measures the targets.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Default (when in scope) | Notes |
|---|---|---|---|
| `PERF-###` targets | `perf-targets` | in | Metric + threshold + load condition + measurement boundary; algorithm+window if a rate limit; sample population/window/re-run rule if a percentile |
| Load profiles | `load-profiles` | in | Expected / peak / growth. For a **real-time/client** product, the load condition is a **workload script** (`SEQ-###`) — see the bar |
| Scaling strategy + capacity model | `scaling-strategy-capacity-model` | in | Horizontal/vertical, capacity per unit |
| Bottleneck analysis | `bottleneck-analysis` | in | Known limits and hotspots |
| Resource budgets | `resource-budgets` | optional | CPU/memory/cost envelopes |

## Owned Contracts (referenced by stable ID)

1. **Performance targets** — `PERF-###`: metric + threshold + load condition + **measurement boundary** (end-to-end vs server-side — or, for client/real-time products, main-thread vs worker-seam vs compositor); where the target is a rate limit, also its **algorithm + window** (e.g. token-bucket, 1s window); where the target is a **percentile**, also its **sample population, window, and re-run rule** (co-owned with Quality 11.5's flake/re-run policy). **Referenced by** Product & Requirements (the NFR target it summarizes), Observability (`SLO-###` measure them), Quality & Testing (perf tests assert them).
2. **Load profiles** — expected/peak/growth; for a frame-budget/real-time product, the **workload script** (`SEQ-###`) that *is* each target's load condition. **Referenced by** Quality (perf-test basis).
3. **Scaling strategy + capacity model.**
4. **Measurement hooks (conditional)** — when Observability (11.11) is out of scope, Performance **owns the measurement-hooks contract**: the marks/counters/probes the product must expose for its `PERF-###` targets to be measured at all (the need doesn't vanish with the concern that usually owns it).

## Consumed (referenced, not owned)

- `CAP-###` (which capabilities carry targets); `COMPONENT-###` (what scales); `PATTERN-###` (Architecture's enabling patterns).

## Contract-grade bar

`PERF-###` targets + load profiles + scaling strategy + capacity model + bottleneck analysis.

- **Each target states its measurement boundary** — where the threshold is measured, since a threshold number alone is ambiguous: **end-to-end** (client-observed, including network) vs **server-side** (handler latency); for client/real-time products the enumeration extends to **main-thread**, **worker-seam**, and **compositor**.
- **A rate-limit target states its algorithm + window, not just the number** — the choice (token-bucket vs fixed-window vs sliding-window) changes the exact edge a test asserts.
- **A percentile target states its sample population, window, and re-run rule** (co-owned with Quality 11.5). The row's **method** details are part of the contract, not harness discretion: how the window is partitioned **including the trailing partial bucket's disposition** (count it, or exclude whole buckets only), and the **sample source** for an "N runs" aggregate (separate harness invocations vs N fresh contexts within one execution).
- **A sequence-shaped load condition states its feasibility assumptions — or delegates route planning explicitly.** For real-time/client products, load collapses into each target via a **workload script** (`SEQ-###`): a contracted input sequence (move here, edit there, hold this key) standing where "requests/second" would. Such a script presumes the product's content makes it executable; the row either states those assumptions or delegates planning to the sanctioned expectation-precomputation pattern (pure product modules in the test process — 11.5).
- **Ownership boundary with Operations:** Performance owns the capacity **envelope** (the `resource-budgets` CPU/memory/cost targets and the capacity model); **Operations owns its realization** (the deployment's actual requests/limits and scaling mechanism).
